# Phase 4 P1: T07 Web UI 鉴权与会话管理实施计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在 T06 基础上完善 Web UI 鉴权能力，建立可恢复、可刷新、可过期治理、可权限门禁且具备基础前端安全防护的会话系统。

**Architecture:** 以 `authStore` 为会话单一事实源（SSOT），`httpClient` 拦截器负责令牌注入与刷新协同，路由层与组件层分别负责认证态守卫与角色/权限门禁。会话策略采用“短生命周期 access token + 可轮换 refresh token（优先 HttpOnly Cookie）”模型，并通过统一 session manager 管理过期、刷新、自动登出和跨标签页同步。

**Tech Stack:** React 18, TypeScript, Zustand (`persist`), React Router v6, Axios interceptors, Vitest + RTL。

## 1. 现状分析（T06 As-Is）

### 1.1 当前实现摘要
- `apps/web/src/features/auth/authStore.ts`
  - 已有状态：`token/memberId/tenantId/user/isAuthenticated`
  - 已有行为：`login/logout/setTenantId`
  - 已启用 `persist`，默认存储为 `localStorage`，key 为 `nexis-auth`
- `apps/web/src/features/auth/LoginPage.tsx`
  - 目前是“手动输入 token/memberId/tenantId”模式
  - 本地校验后直接调用 `login()`
- `apps/web/src/app/routes/ProtectedRoute.tsx`
  - 仅按 `isAuthenticated` 布尔值做路由跳转
- `apps/web/src/shared/api/httpClient.ts`
  - 请求拦截器注入 `Authorization` 与 `X-Tenant-ID`
  - 收到 `401` 时直接 `logout` + `window.location.href='/login'`
- `apps/web/src/shared/ws/wsClient.ts`
  - 使用 URL query 传递 `token`（`?token=...`）

### 1.2 缺口与风险
- 无 refresh token 机制，access token 过期后只能硬登出。
- 无会话到期时间/空闲超时概念，无法实现自动登出策略。
- `ProtectedRoute` 无角色和权限能力，只能“已登录/未登录”二分。
- token 当前默认持久化在 `localStorage`，若出现 XSS 风险面较大。
- WebSocket 通过 query 传 token，存在被日志/代理暴露风险（需后续协议升级）。

## 2. T07 设计方案

### 2.1 会话模型扩展
新增会话域（建议在 `authStore` 或 `sessionStore`）：
- `accessToken: string | null`
- `refreshToken: string | null`（仅在后端暂不支持 HttpOnly 时作为过渡）
- `tokenType: 'Bearer'`
- `expiresAt: number | null`（epoch ms）
- `refreshExpiresAt: number | null`
- `lastActivityAt: number | null`
- `rememberMe: boolean`
- `status: 'anonymous' | 'authenticated' | 'refreshing' | 'expired'`
- `roles: string[]`
- `permissions: string[]`

### 2.2 Token 存储策略
- 推荐策略：
  - `accessToken`：内存 + 可选持久化（由 `rememberMe` 决定）
  - `refreshToken`：目标为 HttpOnly + Secure + SameSite Cookie（前端不读写）
- 前端兼容策略（后端未支持 Cookie 刷新时）：
  - `rememberMe=false` -> `sessionStorage`
  - `rememberMe=true` -> `localStorage`
- `persist` 改造点：使用 `createJSONStorage(() => localStorage/sessionStorage)` 动态选择。
- `partialize`：仅持久化必要字段，避免把临时状态/错误对象写入存储。

### 2.3 Token 刷新机制
- 触发条件：
  - 请求前检测到 `expiresAt - now < refreshThreshold`（例如 60s）
  - 或接口返回 `401 + token_expired` 语义码
- 机制要求：
  - 单飞（single-flight）：并发请求仅触发一次刷新，其余等待同一个 Promise
  - 刷新成功：原请求重放一次
  - 刷新失败：统一会话失效处理并跳转登录
- 兜底：当无 refresh 能力时，保留 T06 行为但增加可观测错误码与用户提示。

### 2.4 会话过期与自动登出
- 绝对过期：`now >= expiresAt` -> 标记 `expired`
- 空闲过期：`now - lastActivityAt > idleTimeout`（例如 30 分钟）
- 自动登出行为：
  - 清空 store 与缓存
  - 关闭 WS 连接
  - 清理定时器
  - 跳转 `/login` 并携带 `reason`（`expired/idle/revoked`）
- 跨标签页同步：监听 `storage` 事件，统一处理登出与会话变更。

### 2.5 权限控制设计
- 路由守卫（认证级）：`ProtectedRoute` 继续负责是否登录。
- 权限门禁（授权级）：新增 `PermissionGate` 组件，支持：
  - `allOf`（必须具备全部权限）
  - `anyOf`（具备任一权限）
  - `fallback`（无权限展示）
- 角色检查：新增 `RoleGate` 或统一到 `AccessGate`，角色来源优先 JWT claims，其次会话接口返回。
- 路由元数据：在 `router.tsx` 中为敏感路由声明 `requiredRoles/requiredPermissions`，由 Guard 统一执行。

### 2.6 安全加固方案
- XSS 防护：
  - 禁止 `dangerouslySetInnerHTML`（如必须使用需 DOMPurify）
  - 输入输出分层：输入校验、输出编码
  - 建议新增 CSP（`script-src 'self'`，限制内联脚本）
- CSRF 防护：
  - 若 refresh 改为 Cookie：采用 `SameSite=Lax/Strict` + CSRF Token（双提交或服务端校验）
  - Axios 统一附带 `X-CSRF-Token`（从 meta 或安全接口注入）
- 敏感数据处理：
  - 禁止日志打印 token、memberId 全量值
  - 错误上报脱敏
  - 登出时清理内存态与持久化态

## 4. 验收标准（T07 Done）
- 会话层支持动态存储策略（`localStorage/sessionStorage`）并可配置。
- access token 可在过期前后自动刷新，且并发场景无重复刷新风暴。
- 会话过期、空闲超时、撤销场景均可自动登出并清理状态。
- 路由级和组件级权限门禁可用，支持角色与权限组合检查。
- 完成 XSS/CSRF/敏感数据三类基础防护并具备测试覆盖。

## 5. 风险与依赖
- 依赖网关提供 refresh 接口与明确错误语义（`expired/revoked`）。
- 若后端暂不支持 HttpOnly refresh cookie，需接受短期 JS 可读 refresh token 风险。
- WebSocket query token 需在后续协议升级中迁移到更安全方案（如握手 header 或一次性 ticket）。
